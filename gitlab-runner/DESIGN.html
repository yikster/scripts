1Problem statement & constraints

We need a GitLab CI runner that builds and deploys infrastructure (AWS CDK in both TypeScript and Python) and triggers CodeBuild projects in multiple other AWS accounts. The runner host is an EC2 instance on Amazon Linux 2023 in a private subnet owned by the security team.

#Constraint (given)Consequence
1Runner runs on EC2 in a private subnetNo public IP, no inbound from internet
2Security-team controlled; no public routingNo IGW route; assume no NAT egress — air-gapped except via PrivateLink + AWS-managed services
3CodeArtifact may be attachedUse as npm + pip/PyPI proxy (upstream fetch is AWS-side)
4An IAM Role may be attachedInstance profile is the identity
5No new IAM UsersAll auth chains from the instance role; no static keys
6The role may be updatedWe can grant exactly what is needed
7Must run CDK (TS + Python) for CIMultiple Node + Python versions on-host
Derived requirement
Every outbound call — AWS API, package download, runner→GitLab — must traverse a VPC endpoint or an AWS-managed service, or it fails.

2Goals and non-goals

Goals

  • Multiple Node (18/20/22) and Python (3.10/3.11/3.12) per job.
  • cdk deploy (TS + Python) and CodeBuild triggers into multiple accounts.
  • No internet egress — packages via CodeArtifact, AWS APIs via VPC endpoints.
  • Instance role only — no IAM users, no static credentials.
  • Idempotent, reproducible (golden AMI), SSH-less (SSM Session Manager).

Non-goals

  • Building the VPC endpoints (network team owns IaC — we specify the list).
  • Hosting GitLab (we only consume it — see the reachability decision).
  • Autoscaling runner fleets (single-host design; templatable later).

End-to-end flow: boot from golden AMI → mise exposes runtimes to the non-login job shell via config.toml environment[] → runner registers with a glrt-* token → each job refreshes the CodeArtifact token and points npm/pip/uv at it → job assumes the target-account role over the regional STS endpoint → cdk bootstrap/deploy or cross-account CodeBuild, all through endpoints.

4Key decision — executor model

Primary recommendation
Shell executor + mise + a host Docker daemon. In an internet-isolated subnet it pulls the fewest images, needs no per-job ECR fetch, and gives CDK direct docker.sock access for asset bundling. mise supplies all Node/Python versions natively.

Alternative: Docker executor + a single pre-baked ECR image (all runtimes + CDK) — only when a team needs strong per-job isolation or high concurrency.

CriterionShell + mise + host Docker (chosen)Docker executor + ECR image
Image pulls in air-gapNone (runtimes on host)One per job (ECR via endpoint)
CDK docker bundlingDirect docker.sockDinD or socket bind
Multi Node/Pythonmise use per jobBaked into the image
Per-job isolationWeaker (shared host)Strong
Operational overheadLowECR image build/scan/lifecycle

The network/security team provisions these on the private subnet.

IMDS note
IMDSv2 (169.254.169.254) is link-local and needs no endpoint, so the instance profile resolves locally — but every sts:AssumeRole still needs the STS interface endpoint.

6IAM design

The instance profile (RunnerRole) is the sole identity — no IAM users, no static keys.

RunnerRole permission policy (iam-runner-policy.json)

  • CodeArtifact read — GetAuthorizationToken (domain), GetRepositoryEndpoint+ReadFromRepository (repos), package read actions (package ARNs).
  • sts:GetServiceBearerToken — Resource: "*" (cannot be scoped) + condition sts:AWSServiceName = codeartifact.amazonaws.com. easy to forget CodeArtifact auth fails without it.
  • Cross-account sts:AssumeRole — Model A cdk-hnb659fds-*-role-<acct>-* and/or Model B OrgDeployRole.
  • Same-account CodeBuild only (cross-account StartBuild is on the assumed target role — see §10).
  • Session Manager, CloudWatch Logs, same-account CloudFormation + EC2 describe, optional SSM Parameter read + S3 cache.

Target-account trust (orgdeployrole-trust.json, Model B) trusts RunnerRole with an sts:ExternalId condition; remove it if unused.

7Runtime management (multiple Node + Python)

Tool: mise — one binary for both Node and Python, fast, per-repo .mise.toml/.tool-versions. Shared at /usr/local/share/mise + /etc/mise; versions Node 18/20/22 and Python 3.10/3.11/3.12 (Python via python-build-standalone, no on-host compiler).

The critical detail
The shell executor runs a non-login bash, which does not source /etc/profile.d/*.sh. Runtimes are injected via config.toml environment[]:
BASH_ENV=/etc/profile.d/mise-shims.sh
MISE_DATA_DIR=/usr/local/share/mise
MISE_CONFIG_DIR=/etc/mise
PATH=<shims first>:...
AWS_STS_REGIONAL_ENDPOINTS=regional

/etc/profile.d/mise.sh is the human-login (SSM) path; mise-shims.sh is the machine path the runner uses. The bootstrap verifies runtimes in the real non-login environment so a green check can't lie.

CDK CLI
Install per-project via npx (pin aws-cdk/aws-cdk-lib in package.json) — never a global npm i -g aws-cdk under multi-Node.

8Packaging — CodeArtifact as npm/PyPI proxy

The instance can't reach npmjs.org or pypi.org. CodeArtifact proxies both; its public upstream fetch happens on the AWS service side, so it works with no instance egress.

  • One domain with npm-store (upstream public:npmjs) and pypi-store (upstream public:pypi).
  • codeartifact-login.sh refreshes the 12-hour token; configures npm (.npmrc), pip (pip.conf, path ends /simple/), uv (named index), optionally poetry/twine.
  • Refresh in the CI before_script every job; a systemd timer is a commented backup.
Secret hygiene
The pip token lives only in pip.conf (chmod 600); PIP_INDEX_URL is deliberately not exported. Never enable CI_DEBUG_TRACE / set -x in before_script.

9GitLab Runner configuration

  • Install: the gitlab-runner binary/RPM needs internet → bake into the golden AMI.
  • Register: modern glrt-* authentication token (legacy registration-token removed in GitLab 18.0). Idempotent on RUNNER_NAME+GITLAB_URL; prunes dead runners via verify --delete.
  • config.toml: concurrent, one [[runners]] with executor=shell, shell=bash, builds_dir/cache_dir under /home/gitlab-runner, and the environment[] runtime injection. Optional S3 cache with ServerAddress=s3.<region>.amazonaws.com (never global).
  • Service: systemd, as the gitlab-runner user (in the docker group).

10Cross-account CDK & CodeBuild

Model A — recommended

cdk bootstrap aws://<target>/<region> --trust <runner-account> --cloudformation-execution-policies <policy> in every target. CDK auto-assumes the target's cdk-hnb659fds-* deploy/publish/lookup roles. RunnerRole only needs sts:AssumeRole on cdk-hnb659fds-*. Re-assumes per operation → sidesteps the 1h STS chaining cap. No profiles needed.

Model B

A dedicated OrgDeployRole per target (trusts RunnerRole, optional ExternalId), used via an ~/.aws/config profile from aws-crossaccount-config.sh with credential_source = Ec2InstanceMetadata + role_arn (no source_profile, no keys).

Cross-account CodeBuild
Cannot be started directly by RunnerRole. The job assumes the target role first, then StartBuild — so codebuild:StartBuild lives on the target OrgDeployRole. The sample pipeline does this in a subshell with a unique role-session-name and a bounded poll loop (≤ ~60 min).

CDK context lookups at synth time also assume the target lookup-role — covered by the Model A wildcard ARNs.

11Golden AMI strategy

In-place install will fail
The bootstrap installers themselves need the internet (mise.run, python-build-standalone, runtime tarballs, the gitlab-runner RPM). Build a golden AMI in an egress-enabled account.
  • Build account: sudo GOLDEN_AMI=false GITLAB_RUNNER_TOKEN= bash init-gitlab-runner-al2023.sh (empty token → bakes everything, does not register). Verify → snapshot (Packer recommended).
  • Never bake a runner token into the image.
  • Launch in the private subnet, attach RunnerRole, IMDSv2 (HttpTokens=required, hop limit 1), no public IP/route. Register over SSM with GOLDEN_AMI=true.

13Open decisions

Hard blocker — decide first
GitLab server reachability. Self-managed in-VPC GitLab (PrivateLink/peering/route) vs GitLab.com SaaS. GitLab.com has no AWS PrivateLink — the runner cannot register/fetch jobs without a security-approved in-VPC proxy or NAT exception. Nothing else works until this is settled.
  1. Golden AMI vs in-place bootstrap — in-place will fail; decide AMI build/versioning ownership (Packer).
  2. CDK trust model A vs B — A simplest + long-deploy safe; B explicit control + ExternalId.
  3. Executor — shell (recommended) vs Docker + ECR image.
  4. CDK in-runner vs CodeBuild trigger — changes which endpoints/IAM are mandatory.
  5. Runner cache backend — S3 distributed vs local-only.
  6. Target account × region matrix — pin to scope ARNs/bootstrap.
  7. ExternalId usage — set consistently across trust/profile/codebuild, or remove.
  8. Secrets source — Secrets Manager (needs endpoint) vs SSM Parameter Store (covered).
  9. Long-job limits — STS chaining 1h; CodeArtifact token 12h.

16Validation & testing

Static (done locally): bash -n on all scripts; JSON parses; sample.gitlab-ci.yml validated with a real YAML parser (Ruby Psych) + an AST duplicate-key / tab / CRLF / type-coercion trap scan — clean. The scanner was verified against a deliberately-broken control file (it fired correctly).

Semantic (run on your side — needs network):

# GitLab CI Lint API
JSON=$(jq -Rs . < gitlab-runner/sample.gitlab-ci.yml)
curl -s -H "PRIVATE-TOKEN: <token>" -H "Content-Type: application/json" \
  --data "{\"content\": $JSON}" "https://<gitlab-host>/api/v4/ci/lint" | jq .
# or Pipeline Editor → Validate, or:  npx gitlab-ci-local --file gitlab-runner/sample.gitlab-ci.yml --list

End-to-end gates: runtime matrix switches versions; npm ci resolves via CodeArtifact; cdk diff does a cross-account lookup; cdk deploy (TS + Python) succeeds; CodeBuild trigger assumes the target role and reaches SUCCEEDED.

17Risks & known limits

  • Chained STS sessions hard-cap at 1h regardless of MaxSessionDuration → prefer Model A per-op re-assume for long deploys; CodeBuild poll bounded under 1h.
  • CodeArtifact token TTL 12h → re-source codeartifact-login.sh mid-job for very long jobs.
  • Token leakage → never enable CI_DEBUG_TRACE/set -x in before_script.
  • uv named index requires the consuming pyproject.toml to declare an index of the same name.
  • Single-runner design — the config.toml patcher targets one named [[runners]] block.
  • GitLab.com SaaS is a hard blocker without an approved proxy/NAT exception.
